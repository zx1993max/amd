## v37-v38 关键发现

### 1. quant位操作优化（v37b，~0.2-0.4μs/shape）
```
替换: tl.log2(amax).floor() → (amax_i32 >> 23) - 129
替换: tl.exp2(-scale)      → ((127 - scale) << 23).to(float32, bitcast)
注意: tl.clamp不支持int32，用tl.where替代
效果: 全shape省0.2-0.4μs，确认有效
```

### 2. waves_per_eu=2（v37b，意外发现）
```
M=64/K=2048: waves_per_eu=4→2 → 14.4→13.2μs（省1.2μs！）
原理: 更少wave = 更多register/wave = 减少register spill
这是调参的重大发现，证明参数搜索仍有空间
```

### 3. BLOCK_SIZE_K=256 + num_stages=2（v38，K loop流水线化）
```
原理: BSK=256比512更小 → 更多K迭代 → num_stages=2可以pipeline(overlap load和compute)
M=16/K=7168: 15.5→14.9μs（-0.6μs，2迭代/split可pipeline）
M=256/K=1536: 18.1(ASM)→16.1(preshuffle)（-2.0μs！preshuffle首次打败ASM）
M=64/K=2048: 13.2→13.1μs（≈same，BSK=256+waves=4 ≈ BSK=512+waves=2）
```

### 4. NUM_KSPLIT>1对非极大K的灾难（v37 教训）
```
M=64/K=2048: NUM_KSPLIT=4 → 14.4→27.1μs（+12.7μs！）
原因: per-tile开销(quant+reshape)无法通过更多tile摊销
      float32输出数据量暴增8x
教训: KSPLIT只对K极大(7168)且每split仅1次K迭代有效
```
